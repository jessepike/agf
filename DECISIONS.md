| 4 | 2026-04-21 | **Seven-layer stack: where control catalogs and risk quantification fit.** AGF sits at Layer 0 as architectural substrate. Layer 1 = OWASP Agentic Top 10 (threat baseline). FAIR and related methodologies are methodology, not primitive — they integrate via AGF's Decision Intelligence capability. | The landscape is crowded. | 5 | 2026-04-21 (refined 2026-04-22) | **Dual-form is load-bearing at gate boundaries.** Refinement of the dual-form principle. Every Gate Resolution and every Domain Outcome MUST emit both a human-readable rationale AND a machine-readable Governance Decision Record (GDR) simultaneously — same decision, two audiences, same moment. "Gate boundary" is defined per #8 (Gate Vocabulary Disambiguation): Gate Resolutions and Domain Outcomes. The GDR (#9) is the canonical machine-form artifact. Affects Primitive #8 (Governance Gates), the Composability Interface spec, and Primitive #11 (Trust Ladders, whose promotions/demotions are Domain Outcomes that emit GDRs). | "Machine form at gates only when needed" (weakens the invariant — once optional, rapidly skipped); "human form as a derived artifact" (breaks OTAA agent-operability if machine form isn't the primary emission); no gate-boundary invariant at all (leaves dual-form as aspirational). |
| 8 | 2026-04-22 | **Gate Vocabulary Disambiguation: four-part vocabulary.** AGF's gate-related vocabularies are NOT alternative spellings of one concept. They form a four-part vocabulary of distinct, scoped enums. **Ring Control Signal** (Composability Interface, `agentic-primitives.md` Composability Interface section): `PASS / REVISE / HALT / GATE / DELEGATE / ERROR` — how primitives in adjacent rings signal each other. **Gate Resolution** (Primitive #8 Governance Gates; also the return value of the `GATE` Ring Control Signal): `APPROVE / REJECT / MODIFY / DEFER / ESCALATE` — the authorizer's response when a gate fires. **Domain Outcome** (domain applications such as Tool Gate): domain-defined enums (e.g., `Authorized / Conditionally Authorized / Denied`) that map onto Gate Resolutions. **Audit Artifact**: GDR (Governance Decision Record) — see entry #9 — the canonical serialized record format for Gate Resolutions and Domain Outcomes. Refines the imprecise "gate decision (PASS/.../ERROR)" wording in earlier prose (#5, internal D12.4a, `relationship-to-frameworks.md`), which conflated Ring Control Signals with Gate Resolutions. The model is unchanged — `agentic-primitives.md` already nests these vocabularies correctly: GATE is a Ring Control Signal that returns a Gate Resolution. | The four vocabularies were each correct in their home doc but were referenced loosely in cross-doc prose, creating apparent ambiguity for readers and blocking schema work for the dual-form gate audit. Naming the four parts explicitly: (a) preserves the existing model (no primitive changes), (b) eliminates the apparent conflict, (c) clarifies what "gate boundary" means for dual-form (#5), (d) makes the GDR primitive's role unambiguous as the canonical audit artifact. Deliberate choice to use names not ordinal numbers in prose to avoid collision with the seven-layer stack (#4) and maturity L1–L5 (#6). | Collapse to one enum (rejected — surfaces serve genuinely different concerns); rename one of the existing enums (rejected — both well-established and accurately scoped); document in shared-vocabulary without a decision entry (rejected — wording bug propagated across DECISIONS.md, internal log, and a public framework doc; needs decision-level visibility); use ordinal labels L1–L4 (rejected — collides with seven-layer stack and maturity levels). |
| 9 | 2026-04-22 | **Governance Decision Record (GDR) is the canonical audit artifact at gate boundaries.** Promotes Q1 from the change queue. The GDR is a reusable AGF primitive — a single record schema with polymorphic content — that serializes any Gate Resolution or Domain Outcome into an auditable artifact. **Scope:** GDR emission required at every Gate Resolution (Primitive #8 fire) and every Domain Outcome (e.g., Tool Gate authorization, Trust Ladder promotion/demotion). Ring Control Signals continue to emit observability events per Primitive #10 (`gate_triggered`, `gate_resolved`, etc.); GDR is the *record* of the decision, observability events are the *log* that the decision occurred. Both are emitted at gate resolution; observability events reference the GDR by `decision_id`. **Schema:** required sections are `metadata`, `decision`, `subject`, `inputs`, `rationale`, `conditions`, `authorization`, `audit`. Canonical schema published as `docs/schemas/gdr.yaml` (structural validation only; lifecycle invariants are normative in the GDR doc prose). **Lifecycle:** GDRs carry `status: pending \| resolved \| expired \| superseded \| aborted`. DEFER is a resolved decision (authorizer actively deferred); a successor GDR with `supersedes` ref captures the eventual real resolution. HALT-on-timeout transitions a pending GDR to `aborted` with the ring-layer HALT signal emitted as observability event. **Sensitive content:** GDRs MUST store classifications, hashes, and access-controlled URIs for sensitive inputs — never raw payloads. **Standards:** GDR provides evidence in support of regulatory record-keeping requirements (EU AI Act Art. 12, NIST AI RMF MANAGE 4.1 logging requirements, ISO 42001 documentation clauses); does not by itself satisfy them. Distinct from an Architectural Decision Record (ADR), which captures engineering design choices. First domain application: Tool Gate (Q3, separate promotion). | Two pressures land on the same artifact: (a) the dual-form gate-boundary requirement (#5) needs a canonical schema target, and (b) the existing grc-profile evidence table already promises "human decision records" for Primitive #8 and "promotion/demotion records" for Primitive #11 without specifying a format. Promoting GDR resolves both with one artifact and makes the grc-profile evidence promises concrete. Naming GDR as a primitive (not "an artifact emitted by Tool Gate") is what makes the format reusable across domains. Scoping to gate boundaries (not every ring signal) preserves observability event volume and matches original D5 intent. | Define separately per primitive (rejected — would create N artifact formats with no cross-primitive audit consistency); leave GDR inside the Tool Gate concept doc (rejected — couples a domain-application doc to the framework's audit primitive); use AICM control evidence schemas as the format (rejected — different artifact class entirely); name it "Governance Decision Artifact" or "Decision Record" (considered — "Record" matches industry usage [ADR, DDR]; "Governance" disambiguates from ADR); broader scope to all Ring Control Signals (rejected per external review — would be a new normative requirement unrelated to D5's original gate-boundary intent and would expand observability storage cost significantly). |

| 7 | 2026-04-21 | **Harness = governance envelope around an AI agent (source + runtime adapters, inseparable).** Formalizes "harness" as an AGF term with the industry-standard meaning. A harness is the declarative source artifacts (intent, constraints, capabilities, policy, evaluation criteria) PLUS the SDK-specific runtime enforcement adapters that translate them into deterministic controls at call time. Formula (industry convention): `Agent = Model + Harness`. Tiered model per AGF Agent Harness Spec v0.3 — Tier 0 (Provider, precondition) / Tier 1 (Agent, maps to AGF Security Fabric) / Tier 2 (Orchestration, maps to AGF Security Governance). Distinct from Wrapper (one deployment mode), Hypervisor (AGT-specific runtime isolation kernel), and Test Harness (general software engineering). | Industry has converged on this definition across Anthropic ("Agent = Model + Harness"), Parallel Web Systems, Philschmid, Atlan, and the Kim & Hwang SSRN paper. AGF's own Agent Harness Spec v0.3 uses the same framing. Earlier internal notes that drew a hard line between design-time scaffolding and runtime enforcement were inconsistent with industry usage and the spec; the Source → Build → Runtime pattern lives *within* the harness, not as a distinction between harness and something else. | Reserve "harness" for design-time scaffolding only (rejected — conflicts with industry usage and AGF's own spec); avoid naming "harness" at all (rejected — it's becoming a load-bearing term in the field; AGF needs a canonical definition); treat "harness" and "wrapper" as synonyms (rejected — wrapper is one deployment pattern; harness is the governance envelope regardless of deployment pattern). |
